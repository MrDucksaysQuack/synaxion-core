# UX 피드백 및 접근성 (Feedback & Accessibility)

**목적**: 모든 사용자 행동에 피드백을 주고, 실패를 숨기거나 UI를 영구 고정하지 않도록 하는 범용 원칙과, 접근성 기본을 정의합니다.

**참고**: 네트워크 타임아웃·fetch 사용 규칙은 [UX_SAFETY_RULES.md](../04-safety-standards/UX_SAFETY_RULES.md)를 따릅니다.

---

## 1. 피드백 원칙

**핵심**: 사용자 경험은 실패할 수 있지만, 실패를 숨기거나 로딩/에러 상태에 영구 고정되면 안 된다.

### 1.1 즉각 피드백 (0–100ms)

- 버튼 호버·포커스·클릭에 시각적 반응(색상·테두리·스케일 등) 제공.

### 1.2 로딩 피드백 (100ms–2s)

- 버튼 클릭 시: 로딩 스피너 또는 비활성화로 중복 클릭 방지.
- 데이터 로딩 시: 스켈레톤 UI 또는 스피너.
- **로딩 상태 고정 방지**: `setLoading(true)` 호출 시 예외·조기 return 경로에서도 반드시 `setLoading(false)` 실행. `try/finally` 또는 동등한 패턴 사용. (상세: [UX_SAFETY_RULES](../04-safety-standards/UX_SAFETY_RULES.md))

### 1.3 결과 피드백 (2s+)

- **성공**: 짧은 메시지·아이콘 또는 Toast. 사용자 언어로 표현.
- **실패**: 에러 메시지·재시도 유도. 기술 용어 대신 사용자 친화적 문구.
- **완료**: Toast 또는 인라인 메시지. 중요한 작업은 모달 등으로 확인 가능하게.

### 1.4 표시 위치

- Toast: 화면 상단·중앙 등 일관된 위치, 자동 소거.
- 인라인: 해당 필드 아래 또는 버튼 옆.
- 모달: 중요한 확인·경고 시.

## 2. 접근성 기본

### 2.1 필수 고려 사항

- **aria-label**: 버튼·아이콘·검색 입력 등 목적이 텍스트로만 드러나는 경우, 스크린 리더용 레이블 제공.
- **키보드**: 포커스 가능 요소는 키보드로 접근·조작 가능. 포커스 순서가 논리적.
- **포커스 표시**: 포커스 링(또는 동등한 시각적 표시)을 제거하지 않음.

### 2.2 페이지·영역

- 페이지 목적을 설명하는 타이틀. 현재 위치(Breadcrumb 등) 명확히.
- 영역(네비게이션, 메인, 보조)은 landmark(role, aria-label 등)로 구분 가능하게.

### 2.3 폼·에러

- 입력 필드에 레이블 연결(label·aria-label·aria-labelledby).
- 검증 에러는 해당 필드와 연결(aria-describedby·aria-invalid 등)하고, 메시지를 사용자 언어로 제공.
