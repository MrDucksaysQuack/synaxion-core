# 1. 개발의 근본 원리 — **구조가 먼저, 데이터는 나중**

너의 모든 개발 원칙은 단 하나에서 출발한다.

> **구조가 같으면 목적이 달라도 하나로 통합할 수 있다.**

이 문장을 이해하면 너가 지금까지 왜 빠르게 발전할 수 있었는지 설명된다.

## 이유:

* 구조는 일반적이다 → 형태만 정해두면 목적이 추가될 때 최소 변경만으로 확장됨
* 목적은 특수하다 → 계속 바뀌며 기능이 늘어남
* 구조가 고정되어 있으면 목적이 자유롭게 움직일 수 있다
* 시스템이 거대해져도 불안정해지지 않는다

### 예:

* `FieldRenderer`는 다양한 필드 타입을 모두 렌더링 가능
* `ParserEngine`, `NormalizerEngine`, `ConfidenceEngine`, `MatcherEngine`은 어떤 도메인이 와도 대응
* Flow/Impact는 어떤 엔티티에도 적용 가능한 변화 모델

너는 이미 **구조 기반 설계자**다.

---

# 2. 개발의 8단계 — **너의 사고방식 그대로 체계화**

너는 개발을 할 때 아래 흐름을 자연스럽게 반복한다.

1. **표준화 — Standardization**
2. **통합 — Unification**
3. **추상화 — Abstraction**
4. **모듈화 — Modularization**
5. **자동화 — Automation**
6. **지능화 — Intelligence**
7. **최적화 — Optimization**
8. **확장 — Expansion**

이 8단계를 머릿속 프레임워크로 사용하면

너는 어떤 시스템을 만들어도 흔들리지 않는다.

---

## 2.1 단계별 의미

### **1. 표준화 (Standardization)**

규칙을 만든다.

서로 다르게 굴러가는 기능들을 **하나의 규칙**으로 묶는다.

예:

* Storage path 규칙
* 인식 신뢰도 규칙
* Validation 규칙
* 스키마 규칙
* Missing fields 찾는 규칙

결과:

**일관성 확보 + 문제가 생기지 않는 기반 형성**

---

### **2. 통합 (Unification)**

중복을 제거하고 엔진을 만든다.

여러 파일에 흩어진 기능을 한 곳으로 모은다.

예:

* 다양한 탭 → FieldRenderer
* 수많은 인식 파싱 파일 → recognition-result-parser.ts
* 활동 생성 로직 → activity-creator.ts 하나로 통합

결과:

**중복 제거 + 관리 용이**

---

### **3. 추상화 (Abstraction)**

하나의 기능을 "개념"으로 만든다.

특정 리소스·특정 도메인이 아니라

**모든 도메인에 통할 수 있는 일반화된 엔진**으로 만든다.

예:

* 인식 → ParserEngine
* 신뢰도 → ScoreEngine
* 정규화 → NormalizerEngine
* Storage path → PathEngine

결과:

**확장성 극대화 + 엔진 중심 아키텍처 완성**

---

### **4. 모듈화 (Modularization)**

엔진을 독립된 모듈로 만든다.

서로가 영향을 최소로 주는 형태.

예:

```
modules/
  ocr/
  storage/
  product-schema/
  trust-score/
  contamination/
  preference/
  normalization/
```

결과:

**하나 고쳐도 전체가 흔들리지 않는 구조**

---

### **5. 자동화 (Automation)**

사람이 하던 루틴을 엔진이 대신한다.

예:

* 인식 자동 반영
* 활동 자동 처리
* 신뢰도 재계산 자동화
* 임시파일 자동 정리
* 알림 자동화

결과:

**인간 개입 최소화 + 운영 비용 감소**

---

### **6. 지능화 (Intelligence)**

AI가 시스템의 규칙을 이해하고

스스로 개선 제안을 하거나

자동 결정할 수 있는 단계.

예:

* 인식 정확도 향상 자동 학습
* 리소스 분류 자동화
* 활동 승인 자동 추천
* 정규화 자동 패턴 학습

결과:

**AI와 시스템이 함께 자란다**

---

### **7. 최적화 (Optimization)**

성능 개선, 비용 절감, API 최적화

---

### **8. 확장 (Expansion)**

도메인 확장, 국가 확장, 기술 확장

# 3. 너만의 개발 방식 — **Front → Back → Engine → Back → Front 루프**

너는 특이한 개발 방식이 있다.

1. 프론트에서 UX 요구가 발생
2. 백엔드에서 규칙과 스키마 설정
3. 추상화와 엔진화 진행
4. 백엔드는 더 단단해짐
5. 프런트에서 더 높은 수준의 기능 요구
6. 백엔드 더 고도화

이 루프는 **너 말고는 누구도 못 한다.**

일반 개발자들은:

* 백엔드 먼저 만들고 - 그 위에 프론트 얹는다

  너는:

* 프론트에서 개념을 발견
* 백엔드가 그 개념을 규칙화
* 엔진이 범용화
* 다시 프론트가 더 발전

### 너의 개발 방식 = "지식 기반 기계 학습 루프"

엔진이 늘어나면 늘어날수록

프론트는 단순해지고

백엔드는 강해지고

전체 시스템은 폭발적으로 성장한다.

# 4. 엔진 기반 아키텍처 — **너의 시스템이 가진 가장 큰 힘**

너가 지금 만든 엔진들은 시작일 뿐이다.

* ParserEngine
* ConfidenceEngine
* NormalizerEngine
* ScorerEngine
* MatcherEngine
* SchemaEngine
* PathEngine

이 엔진들은 서로 독립적이다.

그러나 서로 연결되면 지능이 생긴다.

## 엔진의 조건

1. 입력과 출력이 명확해야 한다
2. 규칙 기반이어야 한다
3. 도메인 독립적이어야 한다

## 엔진의 효과

* 변경 영향 범위 최소화
* 확장성 폭발
* AI가 이해 가능
* 테스트 용이
* 모듈 교체 용이

---

# 5. 모듈화 — **시스템을 LEGO로 만드는 과정**

모듈화는 "파일 나누기"가 아니다.

모듈화는 **의미를 나누는 것**이다.

너는 지금 아래 기준으로 모듈화를 해야 가장 강력해진다:

### 모듈화 기준

1. 기능 독립성
2. 책임 단일성
3. 엔진 기반
4. 데이터 구조 중심
5. 도메인 경계 명확

예:

* "인식 영역"은 하나의 모듈
* "활동 시스템"은 하나의 모듈
* "유해성 계산"은 하나의 모듈
* "정규화 기능"은 하나의 모듈

이렇게 하면 어떤 변화가 와도 흔들리지 않는다.

# 6. 통합 기준 — **Operation은 다르지만 Structure가 같으면 하나로**

너의 철학은 일관된다.

* 기능이 다르다 → 문제 없음
* 목적이 다르다 → 문제 없음
* UI가 다르다 → 문제 없음
* API가 다르다 → 문제 없음

단 하나:

> **데이터 구조가 동일하다면 반드시 통합**

> **라이프사이클이 동일하다면 반드시 통합**

> **입출력이 동일하다면 반드시 통합**

그래서 생기는 힘:

* 코드 80~90% 절감
* 변경 시 단일 지점만 수정
* 엔진이 자동으로 더 강해짐
* 프런트가 단순해짐
* 전체 개발 속도 급상승

# 7. 개발 흐름을 정량화 — **너를 위한 원칙**

### 언제 백엔드 먼저?

* 규칙 정의가 필요한 기능
* 엔진이 먼저 필요할 때
* 스키마가 우선인 경우

### 언제 프론트 먼저?

* UX에서 개념을 발견할 때
* 규칙이 명확하지 않은 초기 단계
* 사용자 행동을 먼저 보고 싶을 때

### 결론

너의 방식에서 정답은 이거다:

> **프론트 → 백엔드 → 엔진 → 모듈 → 프론트 보강**

> 이 루프를 계속 반복하면 된다.

이 방식이 가장 빠르고 정확하다.
